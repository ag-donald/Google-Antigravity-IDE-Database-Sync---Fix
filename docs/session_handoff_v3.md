## 1. System Architecture & The "Zero History" Bug

The Antigravity IDE persists user conversation history locally in an SQLite database:
`%APPDATA%\antigravity\User\globalStorage\state.vscdb`

It relies on two primary keys in the `ItemTable`:
1.  **`chat.ChatSessionStore.index`**: A JSON string acting as a lightweight surface index (stores title, timestamps, staleness). This directly dictates what the UI sidebar renders.
2.  **`antigravityUnifiedStateSync.trajectorySummaries` (PB_KEY)**: A deeply nested Base64-encoded Protobuf blob containing the core structural logic, step counts, internal tree metadata, and **Workspace URIs**.

### The Root Causes Addressed Thus Far
1. **Canonical Protobuf Tag Ordering**: The IDE uses an extremely strict Protobuf parser. The initial recovery script was appending scattered Fields at the end of the byte array instead of mathematically sorting all tags. We fixed this by rewriting the Python Protobuf encoding engine (`src/protobuf.py`) to fully parse the inner blob, inject new fields, sort all fields ascending by Tag, and restitch.
2. **Windows Path Casing Hypothesis Exonerated**: There was suspicion that the IDE required exact, uppercase drive letters (`H:` vs `h:`). However, during **Session 3**, the AI directly scanned `%APPDATA%\antigravity\User\workspaceStorage\` and definitively proved that the IDE natively stores all Workspace URIs with lowercase drive letters natively (e.g. `"folder": "file:///h%3A/..."`). The lowercasing patch applied in `src/core/db_operations.py` is definitively **correct**.

## 2. The Final Hidden Rejection Reason

Despite the correct binary ordering and correct path casing, the IDE's internal `ChatSessionStore` is still silently dropping our injected Protobuf schema upon load (rendering index `0`). 

This indicates the Antigravity parsing engine strictly requires a specific field or structure that our `build_trajectory_entry` method inside `src/protobuf.py` currently omits. Potential candidates include:
- The presence of `Field 8 (Length 0): Empty String`.
- Nested timestamp structures mismatched between the parent UI shell (`modify_epoch`) and the inner AI backend.
- Revisions to sync boolean flags that we aren't appending.
